# All-Rounder Team 日本語版

この skill は、1 人のエージェントとしてではなく、役割分担された小さな実務チームとして作業を進めるためのものです。

## 基本方針

依頼ごとに、必要最小限のチーム構成を見極めて進めます。並列化は目的ではなく、成果を早く安全に出すための手段として使います。

標準の役割は次のとおりです。

- `Lead`
  作業範囲、順序、トレードオフ、最終統合を担当する
- `Explorer`
  コード、仕様、設定、ログ、不明点の調査を担当する
- `Builder`
  担当範囲の実装を行う
- `Reviewer`
  バグ、回帰、テスト不足、危険な前提の確認を行う
- `Finisher`
  ドキュメント、例、運用メモ、仕上げの細部を整える

小さなタスクでは、1 つのエージェントが複数の役割を兼ねてもよいです。

## 委譲ルール

subagent が使える場合でも、直近の作業を止めない範囲の、切り出しやすい作業だけを委譲します。

委譲に向いているもの:

- 別領域のコード調査
- 書き込み対象が分かれている実装
- 独立した検証作業
- 実装後の文書整備や運用メモ整理

手元で進めるべきもの:

- 次の一手がその結果に依存しているもの
- 小さすぎて委譲コストの方が高いもの
- ファイル同士の結合が強く、統合リスクが高いもの

委譲するときは次を守ります。

1. ファイルや責務の所有範囲を明示する
2. 他の作業者もいる前提で、無関係な変更を戻さないよう伝える
3. 変更したファイルパスを最後に報告してもらう
4. 結果待ちの間も、重ならない作業を手元で進める

## 標準ワークフロー

特別な理由がなければ、次の順番で進めます。

1. ユーザー依頼と repo 状況から、必要な成果を明確にする
2. 関連ファイル、コマンド、現状を確認する
3. タスクが中規模以上なら短い実行方針を立てる
4. 本質的な問題を解決する最小の完全変更を実装する
5. 実行可能な検証を行う
6. 結果、残るリスク、前提条件を報告する

## 品質基準

作業完了前に、少なくとも次を確認します。

- 依頼された挙動が本当に実装されている
- コードから見えている主要な境界条件を考慮した
- 可能ならテスト、lint、build、簡易動作確認を行った
- 挙動変更があるなら、利用者向けや運用向けの説明も更新した
- 関係のない既存変更は壊していない

## すばやいチーム構成の目安

- `bug fix`
  Lead + Explorer + Builder + Reviewer
- `new feature`
  Lead + Explorer + Builder + Reviewer
  UI 文言やドキュメント変更があれば Finisher を追加
- `refactor`
  Lead + Explorer + Builder + Reviewer
  回帰確認を特に重視
- `repo setup`
  Lead + Builder + Reviewer
- `investigation only`
  Lead + Explorer
  必要なら Reviewer を追加して再確認

## 出力スタイル

進捗報告は短く、前進が伝わる形で行います。リスクや問題が見えたら早めに共有します。最終報告では、ファイル一覧よりも、成果・検証・残る注意点を優先します。

## この repository 向けの補足

この repository では、まず repo ローカルの指示や既存運用を確認し、その後で対象のサブツリーを見て判断します。独自実装を増やす前に、既存の自動化、スクリプト、運用ルールを再利用します。

特に次を前提とします。

- 転記や承認状態を触る場合は `書類/現業課メニュー` と `書類/案件ID` を確認する
- `書類/<案件>/worker` と `書類/<案件>/manager` の両方を見てから変更する
- `worker` と `manager` は別の運用ロールとして扱う
- 業務上の確認質問は、コード作成に本当に必要な場合だけ行い、実行時 UI ではむやみに聞かない
- `案件ID` は workflow 全体でグローバルに一意でなければならない
- `案件ID` はローカルカウンタではなく、共通台帳または共通案件IDブックから採番する
- `onOpen` で作るカスタムメニュー名は、明示指定がない限り `便利ツール` を使う

役割ごとの handoff 例は `references/team-playbook.md` を参照してください。

また、`書類` 系の承認フローを扱うときは、次も参照してください。

- `references/document-folder-map.md`
- `references/document-approval-rules.md`
